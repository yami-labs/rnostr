Relay-IM-Core 项目文档1. 项目概述与设计意图项目名称Relay-IM-Core项目描述Relay-IM-Core 是一个基于 Nostr 协议构建的极简去中心化即时消息 (IM) 中继服务器。它专注于私人直接消息 (DM) 和群聊房间 (room)，优先考虑隐私、最小化持久存储和去中心化。该项目从 rnostr fork 而来，但进行了深度定制，移除公共 feed 功能，将其简化成以 IM 为导向的系统。设计意图核心意图是创建一个轻量级中继，用于安全、实时的 IM 通信，而无需完整 Nostr 社交功能的开销。主要原则：

- 隐私优先：为 DM 使用端到端加密 (NIP-17)，为房间使用简化的共享密钥加密。最小化元数据暴露和中继端数据保留。
- 最小化持久存储：消息和文件自动过期 (默认 1 年，最短 6 个月)。客户端负责缓存历史记录，中继仅提供临时存储和转发。
- 通过 P2P 实现去中心化：中继之间通过 libp2p mesh 网络发现、转发消息/查询，并支持双备份以提升韧性。
- Web3 集成：使用 SIWE (Sign-In with Ethereum) 进行认证，将用户身份绑定到 Ethereum 地址，以抵抗 sybil 攻击。
- 简洁高效：针对 4 核 8GB 硬件，支持 1000 并发用户。没有公共 feed、搜索或复杂 NIP，仅支持 DM/房间基础功能。
- 目标用例：适用于 DAO、小团队或社区的私人聊天，需要安全、去中心化的消息系统，而无需中心化服务器。
- 非目标：完整 Nostr 兼容性、大规模社交网络、永久存储，或高级功能如 VOIP (未来扩展)。
- 假设：用户是善意的，焦点是 MVP 可用性，并为未来插件 (如 metrics) 留出扩展性。

2\. 系统架构高层架构系统是一个单二进制 Rust 应用，使用 axum 处理 HTTP/WS，libp2p 处理 P2P，LMDB 处理存储。它被设计为库式核心，带有最小 CLI。

- 前端/客户端：不包含 (假设外部 PWA 或应用使用 nostr-tools)。客户端通过 WS 连接，SIWE 认证，发送/接收事件。
- Relay Core：处理 WS 连接、消息路由、存储和 P2P。
- P2P Mesh：中继之间发现、转发和备份。

数据流：

1.  客户端连接 WS → 接收 SIWE challenge → 签名并发送 AUTH。
2.  客户端发送 EVENT (DM/房间消息) → 中继验证 (认证、kind、expire) → 存储到 LMDB → 本地广播或通过 P2P 转发。
3.  客户端发送 REQ (查询历史) → 中继查询 LMDB → 如果无结果，通过 P2P 代理查询 → 返回事件。
4.  GC 任务定期清理过期数据。

组件与模块

- src/main.rs：入口点，加载配置，初始化状态，启动 axum 服务器 (WS + HTTP 端点如 /upload, /metrics, /health)，生成 GC/mesh 任务。
- src/state.rs：AppState (全局共享状态：connections DashMap, broadcast channel, config RwLock, mesh_proxy, room_index)；ConnectionState (每个连接：auth, subscriptions)。
- src/ws.rs：WS 升级处理器，连接循环 (challenge 发送, 消息接收/分发, 广播推送, 清理)。
- src/handler.rs：消息处理器 (handle_event, handle_req 等)，带有 IM 过滤和 P2P 后备。
- src/message.rs：Incoming/OutgoingMessage 结构，解析/序列化。
- src/auth.rs：SIWE 认证器 (challenge 生成, 验证)。
- src/filter.rs：Filter 结构，is_im_compatible/matches 方法。
- src/storage.rs：ImStorage 对 db/ 的封装 (LMDB 操作：save/query/gc)。
- src/db/：内置 LMDB (lmdb.rs, scanner.rs, mod.rs 用于导出)。
- src/gc.rs：事件/文件垃圾回收 (定时任务)。
- src/config.rs：ConfigManager (加载/热重载 config.toml)。
- src/mesh.rs：Libp2pMesh (发现, 双备份, 转发/查询)。
- src/upload.rs：文件上传处理器 (50MB 限制, MIME/扩展名检查, 存储到 blobs/)。
- extensions/metrics/：可选 metrics 插件 (feature-gated, Prometheus 计数器/仪表)。

详细技术规格

- 性能规格：目标负载 1000 并发用户，4 核 8GB 硬件，LMDB 支持 100 万事件级存储，查询延迟 < 100ms (房间索引优化)，消息转发 < 200ms (P2P)。
- 网络规格：WS over HTTP/1.1 或 2 (axum)，TLS 1.3 (rustls, 可配置 cert/key)，P2P: libp2p 0.56 (Kademlia DHT + Gossipsub, TCP + Noise + Yamux)。
- 存储规格：LMDB 嵌入式 KV (内存映射, ACID), 主树 key = event_id, value = json bytes；房间索引树 key = "room_id:created_at:id"；过期通过 tag "expire" 或默认 TTL (1 年)。
- 认证规格：SIWE (EIP-4361), alloy 0.8 验证签名, 支持 EIP-1271 合约钱包；challenge 包含 nonce/domain/uri/chain_id。
- 加密规格：客户端侧, DM: NIP-44 v2 Noise XK (secp256k1), 房间: AES-256-GCM 共享密钥 (定期轮换)。
- 文件规格：上传 POST /upload (multipart), 50MB 上限, MIME 白名单 (image/video/zip/pdf/doc 等), 扩展名黑名单 (exe/bat 等)；存 ./blobs/uuid.ext, 过期随事件 tag "file_expire"。
- 配置规格：toml 文件, 热重载 (notify 监听), 支持 listen_addr, lmdb_path, blob_dir, siwe, gc, tls 等。
- 监控规格：metrics 0.23 (feature), Prometheus /metrics 端点, 指标: active_connections, received_messages, im_events_processed 等。
- 错误规格：thiserror 定义统一 Error, 覆盖 IO/JSON/auth/storage/mesh 等。
- 依赖规格：Rust 1.75+, axum 0.7, libp2p 0.56, siwe 0.6, alloy-primitives 0.8, serde 1.0, tracing 0.1 等 (详见 Cargo.toml)。
- 构建/部署规格：Cargo build (release), Docker 单镜像, 环境变量 RUST_LOG=info。

风险与应对

- 如果 libp2p 开销高，后备使用简单 WebSocket peer 连接。
- 监控 LMDB 规模；超过 100 万事件时添加分片。
- 公开发布前进行安全审计。
